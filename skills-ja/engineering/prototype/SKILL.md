# Prototype

プロトタイプとは **問いに答える使い捨てコード** だ。問いが形を決める。

## 分岐を選ぶ

どの問いに答えているのかを見極める — ユーザーのプロンプトから、周辺コードから、あるいはユーザーが近くにいるなら尋ねて:

- **「このロジック / 状態モデルの感触は正しいか?」** → [LOGIC.md](LOGIC.md)。紙上では推論しづらいケースを state machine に通していく、小さなインタラクティブなターミナルアプリを作る。
- **「これはどう見えるべきか?」** → [UI.md](UI.md)。1 つのルート上に極端に異なる複数の UI バリアントを生成し、URL の search param とフローティングのボトムバーで切り替える。

2 つの分岐はまったく異なる成果物を生む — ここを誤るとプロトタイプ全体が無駄になる。問いが本当に曖昧でユーザーにも連絡がつかないなら、周辺コードによりよく合致する分岐をデフォルトにし（backend の module なら logic、page や component なら UI）、その前提をプロトタイプの冒頭に明記する。

## 両分岐に共通するルール

1. **初日から使い捨てであり、そうと明示する。** プロトタイプのコードは実際に使われる場所の近くに置き（プロトタイプ対象の module や page の隣）、文脈が明白になるようにする — ただし production ではなくプロトタイプだと軽く読んだだけの人にもわかる名前を付ける。使い捨ての UI ルートは、プロジェクトが既に使っている routing 規約に従うこと。新しいトップレベル構造を発明するな。
2. **1 コマンドで実行できる。** プロジェクト既存の task runner がサポートするもの — `pnpm <name>`、`python <path>`、`bun <path>` など。ユーザーが何も考えずに起動できなければならない。
3. **デフォルトでは永続化しない。** 状態はメモリ上に持つ。永続化はプロトタイプが _検証している_ 対象であって、依存すべきものではない。問いが明示的に database を含む場合は、"PROTOTYPE — wipe me" のような明快な名前を付けた scratch DB かローカルファイルを叩く。
4. **磨き込みは省く。** テストなし、プロトタイプを _実行可能_ にする以上の error handling なし、抽象化なし。目的は素早く何かを学ぶことだ。
5. **状態を可視化する。** アクションのたび（logic）またはバリアント切り替えのたび（UI）に、関連する状態全体を print または render し、何が変わったかをユーザーが見られるようにする。
6. **終わったら捕捉する。** 検証済みの判断を実コードに畳み込み、次にプロトタイプそのものを **一次情報 (primary source)** として捕捉する: main の外の使い捨てブランチにコミットし、そのブランチへの context pointer を実装 Issue に残す。答えも捕捉する — 結論とそれが決着させた問いを、Issue かコミットに残す。main ブランチには検証済みの判断だけを残す。
