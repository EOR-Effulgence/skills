# Logic Prototype

ユーザーが手で状態モデルを駆動できる、小さなインタラクティブなターミナルアプリ。問いが **business logic、状態遷移、データ形状** に関するもの — 紙上では妥当に見えるが、実ケースを通してみて初めて違和感が出るもの — に対して使う。

## これが正しい形になるとき

- 「この state machine が X のあと Y が来る edge case を処理できるか自信がない」
- 「このデータモデルで … のケースを本当に表現できるのか」
- 「書く前に API がどうあるべきか触って感じたい」
- ユーザーが **ボタンを押して状態の変化を見たい** もの全般。

問いが「これはどう見えるべきか」なら分岐違い。[UI.md](UI.md) を使う。

## プロセス

### 1. 問いを明文化する

コードを書く前に、どの状態モデルの、どの問いをプロトタイプしているかを書き留める。1 段落でいい。プロトタイプの README か、ファイル先頭のコメントに書く。間違った問いに答える logic プロトタイプは純粋な無駄だ — 後で確認できるよう問いを明示しておく。ユーザーが今見ていようと、AFK で後から戻ってこようと変わらない。

### 2. 言語を選ぶ

ホストプロジェクトが使っているものを使う。プロジェクトに明確な runtime が無い場合（docs リポジトリなど）は尋ねる。

ツーリングはプロジェクトの既存規約に合わせる — プロトタイプのためだけに新しい package manager や runtime を足すな。

### 3. ロジックを移植可能な module に隔離する

実ロジック — 問いに答えている部分 — を、小さく純粋な interface の背後に置き、後で本物のコードベースに持ち上げてそのまま落とし込めるようにする。周りの TUI は使い捨て。ロジック module はそうではない。

正しい形は問いによって変わる:

- **純粋な reducer** — `(state, action) => state`。アクションが離散的なイベントで、状態が単一の値のときに良い。
- **state machine** — 明示的な状態と遷移。「今この瞬間そもそもどのアクションが合法か」が問いの一部のときに良い。
- **平らなデータ型に対する純粋関数の小さなセット**。暗黙の現在状態がなく、ただの変換だけのときに良い。
- **明確なメソッド面を持つクラスや module**。ロジックが本当に継続的な内部状態を所有するとき。

問われている問いに最も合う形を選ぶこと。TUI に繋ぐのが最も楽な形ではない。純粋に保て: I/O なし、ターミナルコードなし、制御フローのための `console.log` なし。TUI がそれを import して呼び込む。逆方向には何も流れない。

これがプロトタイプを自身の寿命を超えて有用にする鍵だ: 問いに答え終わったとき、検証済みの reducer / machine / 関数群を、単体で本物の module に持ち上げられる。

### 4. 状態を露出する最小の TUI を作る

**軽量 TUI** として作る — 毎ティック画面をクリアし（`console.clear()` / `print("\033[2J\033[H")` / 同等品）、フレーム全体を再描画する。ユーザーには常に 1 つの安定したビューを見せ、際限なく膨らむスクロールバックを見せるな。

各フレームは次の順序の 2 部構成:

1. **現在の状態**。見やすく diff フレンドリーに（1 行 1 フィールド、または整形済み JSON）。フィールド名やセクション見出しには **bold**、重要度の低い文脈（timestamp、ID、派生値）には **dim** を使う。生の ANSI エスケープコードで構わない — `\x1b[1m` が bold、`\x1b[2m` が dim、`\x1b[0m` が reset。プロジェクトに既に入っているのでなければ、styling library を持ち込む必要はない。
2. **キーボードショートカット**。下部に列挙する: `[a] add user  [d] delete user  [t] tick clock  [q] quit`。キーを bold、説明を dim、あるいはその逆 — きれいに読める方で。

挙動:

1. **状態を初期化する** — メモリ上の単一のオブジェクト / 構造体。起動時に最初のフレームを描画する。
2. **キーストローク（または 1 行）を 1 つずつ読み取り**、状態を変更するハンドラへ dispatch する。
3. アクションのたびに **フレーム全体を再描画する** — 追記ではなく差し替え。
4. **quit まで loop する。**

フレーム全体は 1 画面に収まること。

### 5. 1 コマンドで実行可能にする

プロジェクト既存の task runner（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）にスクリプトを足す。ユーザーは `pnpm run <prototype-name>` か同等品で実行できるべきだ — パスを覚える必要は決してあってはならない。

ホストプロジェクトに task runner が無いなら、コマンドをプロトタイプの README 先頭に書いておく。

### 6. 引き渡す

ユーザーに実行コマンドを渡す。あとは彼ら自身が駆動する。面白い瞬間は「待って、これは起こりえないはずだ」「あれ、X はこうじゃないと思ってた」と言うときだ — それが _アイデア_ そのもののバグであり、それこそが全ての要点だ。新しいアクションを足したいと言われたら足す。プロトタイプは進化する。

### 7. 答えとプロトタイプを捕捉する

プロトタイプが問いに答え終わったら、まず答えを捕捉し、次に [SKILL](SKILL.md) が述べるやり方でプロトタイプを捕捉する。logic 固有の対応付けはこうだ: 検証済みの reducer / machine / 関数群は本物の module へ持ち上がる（判断が吸収される）。TUI シェルは、プロトタイプを一次情報 (primary source) として保つ使い捨てブランチへ一緒に乗っていく。

## アンチパターン

- **テストを足すな。** テストを必要とするプロトタイプはもうプロトタイプではない。
- **本物の database に繋ぐな。** 問いが明示的に永続化についてでない限り、インメモリのストアを使う。
- **一般化するな。** 「あとで X をサポートしたくなったら」は無し。プロトタイプは 1 つの問いに答える。
- **ロジックと TUI を混ぜてぼかすな。** reducer / state machine が `console.log`、prompt、ターミナルエスケープコードを参照していたら、もう移植可能ではない。TUI は純粋な module の上の薄いシェルに留めろ。
- **TUI シェルを production に出荷するな。** シェルはターミナルから手で駆動されることに最適化されている。残す価値があるのは、その背後にあるロジック module の方だ。
