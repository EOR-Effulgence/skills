# UI Prototype

**極端に異なる UI バリアントを複数**1 つのルート上に生成し、フローティングのボトムバーから切り替えられるようにする。ユーザーはブラウザでバリアントを行き来し、1 つを選ぶか（または各バリアントから良いところを盗み出して）、残りを捨てる。

問いがロジック / 状態に関するもので、見た目ではない場合は分岐違い。[LOGIC.md](LOGIC.md) を使う。

## これが正しい形状になるとき

- 「このページはどう見えるべきか？」
- 「ダッシュボードのオプションをいくつか見てから決めたい」
- 「設定画面のレイアウトを別のものに試してみて」
- 頭の中の 3 つの曖昧なモックアップから選ぶのに丸一日使うようなとき。

## 2 つのサブ形状 — サブ形状 A を強く推奨

UI プロトタイプは、**アプリの他の部分と隣接した状態**で判断する方がはるかに楽だ — 本物のヘッダー、本物のサイドバー、本物のデータ、本物の密度の中で。単独の使い捨てルートは真空地帯であり、どのバリアントも単独だと「なんとなく良さそう」に見える。バリアントを宿せる既存ページが妥当にある場合は必ずサブ形状 A をデフォルトにする。プロトタイプが本当に近場に住処を持たない場合のみサブ形状 B に逃げる。

### サブ形状 A — 既存ページへの調整（推奨）

ルートはすでに存在する。バリアントは**同じルート上で**レンダリングされ、`?variant=` URL クエリパラメータでゲートされる。既存のデータ取得、パラメータ、認証はすべて維持され、レンダリング部分だけが入れ替わる。これがデフォルト — 特別な理由がない限りこれを選ぶ。

プロトタイプ対象がまだページを持たないが、**自然に既存ページの中に住むはずのもの**（ダッシュボードの新セクション、設定画面の新カード、既存フローの新ステップ）であれば、それも依然サブ形状 A。ホストページの中にバリアントをマウントする。

### サブ形状 B — 新規ページ（最終手段）

プロトタイプ対象が本当に住める既存ページを持たないとき — まったく新しいトップレベル面、またはどこにも自然に埋め込めないフローのとき — のみ使う。

プロジェクトの既存ルーティング規約に従って**使い捨てルート**を作る — 新しいトップレベル構造を発明しない。明らかにプロトタイプだと分かる名前を付ける（パスやファイル名に `prototype` を含めるなど）。同じ `?variant=` パターン。

サブ形状 B にコミットする前に冷静になれ: 本当に埋め込める既存ページは無いのか？ 空のルートは、人が住んでいるルートなら露呈するはずの設計問題を隠す。

両サブ形状でフローティングのボトムバーは同一。

## プロセス

### 1. 問いを明文化し、N を決める

デフォルトは**3 バリアント**。5 を超えると「極端に異なる」ではなくただのノイズになる — 5 でキャップ。

計画を 1 行、プロトタイプの場所またはファイル先頭コメントに書く:

> 「設定ページの 3 バリアントを `?variant=` で切り替え、既存の `/settings` ルート上に置く」

ユーザーが押し返しに来ようと来まいと、これで機能する。

### 2. 極端に異なるバリアントを生成する

各バリアントを下書きする。各バリアントは次に対して保たれること:

- ページの目的とアクセスできるデータ
- プロジェクトのコンポーネントライブラリ / スタイリングシステム（TailwindCSS、shadcn、MUI、生 CSS、何でも）
- 明確にエクスポートされたコンポーネント名、例: `VariantA`、`VariantB`、`VariantC`

バリアントは**構造的に異なる**こと — 異なるレイアウト、異なる情報階層、異なる主要アフォーダンス。色違いではない。微妙に違うだけのカードグリッド 3 つは UI プロトタイプではなく壁紙だ。下書きが似てしまったら、「カードグリッドは使うな」と明示してやり直す。

### 3. 配線する

ルート上に単一のスイッチャーコンポーネントを作る:

```tsx
// 疑似コード — プロジェクトのフレームワークに合わせて
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

サブ形状 A（既存ページ）の場合: 既存のデータ取得をスイッチャーの上に置いたまま保持し、変わるのはレンダリングされるサブツリーだけにする。

サブ形状 B（新規ページ）の場合: `/prototype/<name>` 下の使い捨てルートが同じスイッチャーをマウントする。

### 4. フローティングスイッチャーを作る

画面下中央に固定された小さなバー。3 つの要素:

- **左矢印** — 前のバリアントへ循環（末尾で先頭に戻る）
- **バリアントラベル** — 現在のバリアントキーと、バリアントが名前をエクスポートしていればその名前も。例: `B — Sidebar layout`
- **右矢印** — 次へ循環（末尾で先頭に戻る）

挙動:

- 矢印クリックで URL クエリパラメータが更新される（フレームワークのルーターを使う — Next なら `router.replace`、React Router なら `navigate` など）。これでバリアントは共有可能かつリロード耐性を持つ
- キーボード: `←` と `→` キーでも循環。`<input>`、`<textarea>`、`[contenteditable]` にフォーカスがあるときは矢印キーを横取りしない
- ページから視覚的に区別される（高コントラストの pill、控えめなシャドウなど）— 評価対象の設計の一部でないことが一目で分かるように
- 本番ビルドでは隠す — `process.env.NODE_ENV !== 'production'` か同等のチェックでゲートし、プロトタイプが事故でマージされてもユーザーにバーが届かないようにする

スイッチャーは単一の共有コンポーネントに置き、両サブ形状から再利用する。プロジェクトで共有 UI が置かれる場所に配置する。

### 5. 引き渡す

URL（および `?variant=` キー）を表に出す。ユーザーは時間ができたら順に切り替えていく。面白いフィードバックは大抵**「B のヘッダーに C のサイドバーが欲しい」** — それが彼らが本当に欲しい設計だ。

### 6. 答えを記録して掃除する

勝者が決まったら、どれが・なぜ勝ったかを書き留める（コミットメッセージ、ADR、Issue、または AFK 実行中でユーザー応答待ちならプロトタイプの隣の `NOTES.md`）。その後:

- **サブ形状 A** — 敗者バリアントとスイッチャーを削除し、勝者を既存ページに畳み込む
- **サブ形状 B** — 勝者を本物のルートへ昇格し、使い捨てルートとスイッチャーを削除する

バリアントコンポーネントやスイッチャーを残すな。すぐ腐り、次の読み手を混乱させる。

## アンチパターン

- **色やコピーしか違わないバリアント。** それは微調整であってプロトタイプではない。本物のバリアントは構造で意見が割れる。
- **バリアント間でコードを共有しすぎる。** 共有 `<Header>` は良し、共有 `<Layout>` は要点を潰す。各バリアントはレイアウトを丸ごと捨てる自由を持つこと。
- **バリアントを本物の mutation に繋ぐ。** 読み取り専用プロトタイプで十分。バリアントが mutation を必要とするなら、スタブに向ける — 問いは「どう見えるべきか」であって「バックエンドは動くか」ではない。
- **プロトタイプを直接本番へ昇格する。** バリアントコードはプロトタイプの制約（テストなし、最小限のエラーハンドリング）で書かれている。畳み込むときは正しく書き直すこと。
