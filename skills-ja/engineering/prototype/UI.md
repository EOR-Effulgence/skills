# UI Prototype

**極端に異なる複数の UI バリアント** を 1 つのルート上に生成し、フローティングのボトムバーから切り替えられるようにする。ユーザーはブラウザでバリアントを行き来し、1 つを選ぶ（または各バリアントから良いところを盗む）と、残りは捨てる。

問いがロジック / 状態に関するもので、見た目についてでないなら分岐違い。[LOGIC.md](LOGIC.md) を使う。

## これが正しい形になるとき

- 「このページはどう見えるべきか?」
- 「決める前にこのダッシュボードの選択肢をいくつか見たい」
- 「設定画面のレイアウトを別のもので試してみて」
- ユーザーが放っておけば、頭の中の 3 つの曖昧なモックアップから選ぶのに丸一日を費やしてしまうようなとき。

## 2 つのサブ形状 — サブ形状 A を強く推奨

UI プロトタイプは、**アプリの他の部分と突き合わせた状態** で判断する方がはるかに楽だ — 本物のヘッダー、本物のサイドバー、本物のデータ、本物の密度。単独の使い捨てルートは真空地帯であり、どのバリアントも孤立して見れば良さそうに見えてしまう。バリアントを宿せる妥当な既存ページがあるときは常にサブ形状 A をデフォルトにする。プロトタイプが本当に近場に住処を持たないときだけ、サブ形状 B に手を伸ばす。

### サブ形状 A — 既存ページへの調整（推奨）

ルートはすでに存在する。バリアントは **同じルート上で** レンダリングされ、`?variant=` URL の search param でゲートされる。既存のデータ取得、params、auth はすべてそのまま — レンダリングだけが入れ替わる。これがデフォルトだ。そうしない特定の理由がない限りこれを選ぶ。

プロトタイプ対象がまだページを持たないが、**自然に既存ページの中に住むはずのもの**（ダッシュボードの新しいセクション、設定画面の新しいカード、既存フローの新しいステップ）なら、それも依然サブ形状 A だ。ホストページの中にバリアントをマウントする。

### サブ形状 B — 新しいページ（最終手段）

プロトタイプ対象が本当に住める既存ページを持たないとき — 例えばまったく新しいトップレベルの面や、どこにも妥当に埋め込めないフロー — のみこれを使う。

プロジェクトが既に使っている routing 規約に従って **使い捨てルート** を作る — 新しいトップレベル構造を発明するな。明らかにプロトタイプだと分かる名前を付ける（例: パスかファイル名に `prototype` という語を含める）。同じ `?variant=` パターン。

サブ形状 B にコミットする前に冷静に確認しろ: これを埋め込める既存ページは本当に無いのか? 空のルートは、人が住むルートなら露呈するはずの設計上の問題を隠す。

どちらのサブ形状でもフローティングのボトムバーは同一だ。

## プロセス

### 1. 問いを明文化し、N を決める

デフォルトは **3 バリアント**。5 を超えると「極端に異なる」であることをやめてノイズになり始める — そこでキャップする。

計画を 1 行で、プロトタイプの置き場所かファイル先頭のコメントに書く:

> 「設定ページの 3 バリアントを、`?variant=` で切り替え、既存の `/settings` ルート上に置く。」

ユーザーが押し返しに来ようと来まいと、これは機能する。

### 2. 極端に異なるバリアントを生成する

各バリアントを下書きする。各バリアントを次の基準に照らす:

- ページの目的と、アクセスできるデータ。
- プロジェクトの component library / styling system（TailwindCSS、shadcn、MUI、生の CSS、何でも）。
- 明確にエクスポートされた component 名、例: `VariantA`、`VariantB`、`VariantC`。

バリアントは **構造的に異なる** こと — 異なるレイアウト、異なる情報階層、異なる主要なアフォーダンス。ただの色違いではない。少しずつ手を加えただけのカードグリッドが 3 つでは UI プロトタイプではなく、壁紙だ。2 つの下書きが似すぎて出てきたら、片方を「カードグリッドは使うな」と明示的に指示してやり直す。

### 3. 配線してつなぐ

ルート上に単一の switcher component を作る:

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

サブ形状 A（既存ページ）の場合: 既存のデータ取得はすべて switcher の上に置いたまま保つ。バリアントごとに変わるのはレンダリングされる subtree だけだ。

サブ形状 B（新しいページ）の場合: `/prototype/<name>` 下の使い捨てルートが同じ switcher をマウントする。

### 4. フローティングの switcher を作る

画面の下中央に固定配置される小さなバー。3 つの部品を持つ:

- **左矢印** — 前のバリアントへ循環する（末尾で先頭に戻る）。
- **バリアントラベル** — 現在のバリアントキーを表示し、バリアントが名前をエクスポートしていればその名前も。例: `B — Sidebar layout`。
- **右矢印** — 前方へ循環する（末尾で先頭に戻る）。

挙動:

- 矢印をクリックすると URL の search param が更新される（フレームワークの router を使う — Next なら `router.replace`、React Router なら `navigate` など）。これでバリアントは共有可能かつリロード耐性を持つ。
- キーボード: `←` と `→` の矢印キーでも循環する。`<input>`、`<textarea>`、`[contenteditable]` にフォーカスがあるときは矢印キーを横取りするな。
- ページから視覚的に区別する（例: 高コントラストの pill、控えめな shadow）。評価対象の設計の一部でないことが一目で分かるように。
- production ビルドでは隠す — `process.env.NODE_ENV !== 'production'` か同等のチェックでゲートし、はぐれたプロトタイプの merge がバーをユーザーに出荷してしまえないようにする。

switcher は単一の共有 component に置き、両サブ形状が再利用できるようにする。プロジェクトで共有 UI が置かれる場所に配置する。

### 5. 引き渡す

URL（と `?variant=` のキー）を表に出す。ユーザーは手が空いたときに順に切り替えていく。面白いフィードバックは大抵 **「B のヘッダーに C のサイドバーが欲しい」** だ — それが彼らが本当に欲しい設計だ。

### 6. 答えを捕捉して片付ける

あるバリアントが勝ったら、まず答えを捕捉する — どのバリアントが、なぜ勝ったか — 次に [SKILL](SKILL.md) が述べるやり方でプロトタイプを捕捉する。勝者を実コードに畳み込み、残りは main ではなく使い捨てブランチへ移す:

- **サブ形状 A** — 勝者を既存ページに畳み込む。負けたバリアントと switcher は main から落とす。
- **サブ形状 B** — 勝ったバリアントを本物のルートへ昇格する。使い捨てルートと switcher は main から落とす。

バリアント一式が一次情報 (primary source) なので、それはゴミ箱ではなく使い捨てブランチに着地する — main ブランチに残されたバリアント component と switcher は素早く腐り、次の読み手を混乱させる。

## アンチパターン

- **色やコピーしか違わないバリアント。** それは微調整であってプロトタイプではない。本物のバリアントは構造について意見が割れる。
- **バリアント間でコードを共有しすぎる。** 共有 `<Header>` は良い。共有 `<Layout>` は要点を潰す。各バリアントはレイアウトを丸ごと投げ捨てる自由を持つべきだ。
- **バリアントを本物の mutation に配線する。** read-only なプロトタイプで十分。バリアントが変更を必要とするなら、スタブに向けろ — 問いは「これはどう見えるべきか」であって「バックエンドは動くか」ではない。
- **プロトタイプを直接 production へ昇格する。** バリアントコードはプロトタイプの制約（テストなし、最小限の error handling）のもとで書かれた。畳み込むときは適切に書き直せ。
