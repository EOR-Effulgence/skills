# Triage

プロジェクトの Issue トラッカー上で、トリアージロール群からなる小さなステートマシンを通じて Issue を動かす。

トリアージ中に Issue トラッカーへ投稿するコメント・Issue は、**必ず**この免責で始めること:

```
> *This was generated by AI during triage.*
```

## 参考ドキュメント

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 耐久性のあるエージェントブリーフの書き方
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` ナレッジベースの仕組み

## ロール

**カテゴリ**ロール 2 種:

- `bug` — 何かが壊れている
- `enhancement` — 新機能 / 改善

**ステート**ロール 5 種:

- `needs-triage` — メンテナの評価待ち
- `needs-info` — レポーターからの追加情報待ち
- `ready-for-agent` — 仕様確定、AFK エージェントに渡せる
- `ready-for-human` — 人間による実装が必要
- `wontfix` — 対応しない

トリアージ済みの Issue は必ずカテゴリロール 1 つとステートロール 1 つを持つ。ステートロールが衝突したら、何か手を動かす前にメンテナに確認しろ。

これらは正典のロール名であって、Issue トラッカー上の実際のラベル文字列は異なる場合がある。マッピングは事前に渡されているはず。なければ `/setup-matt-pocock-skills` を実行する。

ステート遷移: ラベル無しの Issue は通常まず `needs-triage` に入る。そこから `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` のいずれかへ動く。`needs-info` はレポーターが返信した時点で `needs-triage` に戻る。メンテナはいつでも上書きしてよい — 不自然な遷移は前に確認を入れること。

## 起動

メンテナは `/triage` を呼び、欲しいことを自然言語で説明する。それを解釈して動け。例:

- 「自分の注意が必要なものを見せて」
- 「#42 を見よう」
- 「#42 を ready-for-agent に動かして」
- 「エージェントが拾える状態のものは？」

## 注意すべきものを見せる

Issue トラッカーをクエリし、古い順に 3 バケットで提示する:

1. **Unlabeled** — 一度もトリアージされていない
2. **`needs-triage`** — 評価中
3. **前回トリアージノート以降にレポーター活動がある `needs-info`** — 再評価が必要

件数と Issue ごとの 1 行サマリを示し、メンテナに選ばせる。

## 特定の Issue をトリアージする

1. **コンテキストを集める。** Issue 全体（本文、コメント、ラベル、レポーター、日付）を読む。過去のトリアージノートを解析し、解決済みの問いを再度ぶつけないようにする。プロジェクトのドメイン用語集を使ってコードベースを探索し、その領域の ADR を尊重する。`.out-of-scope/*.md` を読み、今回の Issue に似た過去の却下を表面化する。

2. **推奨する。** カテゴリとステートの推奨を理由付きでメンテナに伝え、Issue に関連するコードベースの簡潔なサマリも添える。指示を待つ。

3. **再現する（bug のみ）。** 詰める前にまず再現を試みろ: レポーターの手順を読み、関連コードを追い、テストやコマンドを走らせる。結果を報告する — コード経路まで含めた再現成功、再現失敗、または情報不足（強い `needs-info` シグナル）。再現が確認できたエージェントブリーフは桁違いに強い。

4. **詰める（必要なら）。** Issue を肉付けする必要があれば `/grill-with-docs` セッションを走らせる。

5. **結果を適用する:**
   - `ready-for-agent` — エージェントブリーフのコメントを投稿（[AGENT-BRIEF.md](AGENT-BRIEF.md)）
   - `ready-for-human` — エージェントブリーフと同じ構造だが、なぜ委譲できないか（判断、外部アクセス、設計判断、手動テスト）を書く
   - `needs-info` — トリアージノートを投稿（下のテンプレート）
   - `wontfix`（bug）— 丁寧な説明、その後クローズ
   - `wontfix`（enhancement）— `.out-of-scope/` に書き、そこへのリンクをコメントから張り、クローズ（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）
   - `needs-triage` — ロールを付与。途中まで進んでいる場合は任意でコメントを残す

## ステートのクイック上書き

メンテナが「#42 を ready-for-agent に動かして」と言ったら、信用してロールを直接当てろ。これからやること（ロール変更、コメント、クローズ）を確認してから動く。詰めはスキップ。詰めセッションを通さずに `ready-for-agent` に動かす場合、エージェントブリーフを書くか確認する。

## Needs-info テンプレート

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

詰めセッションで解決したことはすべて "established so far" にまとめ、作業を取りこぼさない。質問は具体的かつ実行可能でなければならない。「もっと情報をください」は禁止。

## 前回セッションの再開

Issue に前回のトリアージノートが残っていたら、それを読み、レポーターが未解決の問いに答えているかを確認し、現状を更新したうえで続行する。解決済みの問いを再度ぶつけない。
