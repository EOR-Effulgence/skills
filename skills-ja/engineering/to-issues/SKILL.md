# To Issues

計画をトレーサー弾（vertical slices）で独立に着手可能な issue 群へ分割する。

issue tracker と triage ラベルの語彙は事前に提示されているはずだ — もし無ければ `/setup-matt-pocock-skills` を走らせる。

## Process

### 1. コンテキストを集める

すでに会話コンテキストに入っているものを土台に作業する。ユーザーが引数として issue 参照（issue 番号、URL、パス）を渡してきたら、issue tracker から取得して本文とコメントを全部読む。

### 2. コードベースを探索（任意）

まだコードベースを探索していなければ、現状を理解するために探索する。issue のタイトルと説明はプロジェクトの**ドメイン用語集**の語彙を使い、触る領域の ADR を尊重する。

### 3. 垂直スライスを起案

計画を**トレーサー弾**の issue 群に分解する。各 issue は全統合レイヤを端から端まで貫く薄い垂直スライス。**ひとつのレイヤだけの水平スライスにしない。**

スライスは「HITL」または「AFK」のどちらか。HITL スライスはアーキテクチャ判断やデザインレビューなど人間の介在を必要とする。AFK スライスは人間の介在なしで実装・マージできる。可能なかぎり HITL より AFK を優先する。

<vertical-slice-rules>
- 各スライスは狭くても、全レイヤ（schema・API・UI・tests）を端から端まで貫く COMPLETE なパスを届ける
- 完了したスライスは単体でデモまたは検証ができる
- 厚いスライスを少数より、薄いスライスを多数を選ぶ
</vertical-slice-rules>

### 4. ユーザーに確認する

提案する分解を番号付きリストで提示する。各スライスについて以下を示す:

- **Title**: 短く説明的な名前
- **Type**: HITL / AFK
- **Blocked by**: 先に完了させるべき他スライス（あれば）
- **User stories covered**: このスライスが対応するユーザーストーリー（元資料にあれば）

ユーザーに問う:

- 粒度は適切か？（粗すぎる / 細かすぎる）
- 依存関係は正しいか？
- マージや更なる分割をすべきスライスはあるか？
- HITL / AFK の振り分けは正しいか？

ユーザーが分解を承認するまで反復する。

### 5. issue tracker に公開

承認された各スライスを issue tracker に新規 issue として公開する。以下の issue 本文テンプレートを使う。これらの issue は AFK エージェントが着手可能とみなされるため、指示がない限り適切な triage ラベルを付けて公開する。

依存順（ブロッカーが先）に公開する。そうすれば「Blocked by」フィールドに実在の issue 識別子を参照できる。

<issue-template>
## Parent

issue tracker 上の親 issue への参照（元資料が既存 issue だった場合のみ。そうでなければこのセクションは省略）。

## What to build

この垂直スライスの簡潔な説明。レイヤ単位の実装ではなく、端から端までのふるまいを記述する。

具体的なファイルパスやコードスニペットは避ける — 古びるのが早い。例外: プロトタイプが、散文より正確に意思決定をエンコードするスニペット（state machine、reducer、schema、type shape）を生んだ場合、それをここにインラインで貼り、プロトタイプ由来であることを簡潔に明記する。意思決定の核となる部分だけに刈り込む — 動くデモではなく、重要な要点だけ。

## Acceptance criteria

- [ ] 基準 1
- [ ] 基準 2
- [ ] 基準 3

## Blocked by

- ブロックしているチケットへの参照（あれば）

ブロッカーがなければ「None - can start immediately」と書く。

</issue-template>

親 issue を close したり変更したりしてはいけない。
