## ドメイン認識

コードベース探索中に、既存のドキュメントも探す:

### ファイル構造

ほとんどのリポジトリはコンテキストが 1 つ:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

ルートに `CONTEXT-MAP.md` があれば、複数コンテキストのリポジトリ。マップは各コンテキストの場所を指す:

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← システム横断の決定
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← コンテキスト固有の決定
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

ファイルは**遅延作成**せよ — 書く中身ができたときだけ作る。`CONTEXT.md` がなければ、最初の用語が固まったときに作る。`docs/adr/` がなければ、最初の ADR が必要になったときに作る。

## セッション中

### 用語集に照らして異議を唱える

ユーザーが `CONTEXT.md` の既存言語と矛盾する用語を使ったら、即座に指摘する。「用語集では『キャンセル』は X と定義されていますが、あなたは Y を意味しているように見えます — どちらですか？」

### あいまいな言葉を研ぎ澄ます

ユーザーがあいまい・過負荷な用語を使ったら、正確な正典用語を提案する。「『アカウント』と言っていますが、Customer ですか User ですか？ 別物です」

### 具体的シナリオで議論

ドメイン関係を議論するときは、具体的シナリオで圧力テストする。エッジケースを突くシナリオをでっち上げ、概念間の境界をユーザーに正確に言わせる。

### コードと相互参照

ユーザーが「こう動く」と言ったら、コードが同意しているか確認する。矛盾を見つけたら表面化する: 「あなたのコードは Order を丸ごとキャンセルしますが、部分キャンセルが可能だと言いました — どちらが正しい？」

### CONTEXT.md をその場で更新

用語が固まったら、`CONTEXT.md` をその場で更新する。**まとめてやらない** — 起きたそばから捕まえる。フォーマットは [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) を使う。

`CONTEXT.md` は**実装の詳細を一切含めない**。仕様書、スクラッチパッド、実装決定の置き場として扱ってはいけない。**用語集であってそれ以外ではない。**

### ADR は控えめに提案する

ADR の作成を提案するのは、以下の **3 つすべて** が真のときだけ:

1. **後戻りしづらい** — 後で気が変わったときのコストが意味のある大きさ
2. **文脈なしでは驚かされる** — 将来の読者が「なぜこうしたのか？」と思う
3. **本物のトレードオフの結果** — 真の代替案があり、特定の理由で選んだ

3 つのうちどれかが欠ければ、ADR は省略する。フォーマットは [ADR-FORMAT.md](./ADR-FORMAT.md) を使う。
