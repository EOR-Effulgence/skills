# Diagnose

難しいバグのための規律。明示的な正当化なしにフェーズをスキップしてはいけない。

コードベースを探索する際は、プロジェクトのドメイン用語集を使って関連 Module の明確なメンタルモデルを作り、触る領域の ADR を確認する。

## Phase 1 — フィードバックループを作る

**ここがこのスキルの本体だ。** ほかは全部メカニカルでしかない。バグに対する高速・決定論的・エージェントが実行可能な pass/fail シグナルを持っているなら、原因は必ず見つかる — bisection、仮説検証、計装はすべてそのシグナルを消費しているだけだ。シグナルを持っていないなら、コードをいくら睨んでも救われない。

ここに不釣り合いなほど労力を注ぐ。**攻める。創造的になる。あきらめない。**

### 作り方 — おおむねこの順で試す

1. バグに届く Seam での**失敗するテスト** — unit、integration、e2e
2. 動いている dev サーバーに対する **curl / HTTP スクリプト**
3. fixture 入力を渡す **CLI 起動** + 既知の良い snapshot との stdout diff
4. **ヘッドレスブラウザスクリプト**（Playwright / Puppeteer）— UI を駆動し、DOM / コンソール / ネットワークに assert
5. **キャプチャしたトレースをリプレイ**。実ネットワークリクエスト / ペイロード / イベントログをディスクに保存し、独立にコード経路へ流し直す
6. **使い捨てハーネス**。システムの最小サブセット（1 サービス、依存はモック）を立ち上げ、関数 1 回呼ぶだけでバグのコード経路を踏むようにする
7. **property / fuzz ループ**。バグが「たまに間違った出力」型なら、ランダム入力 1000 件を流して failure mode を探す
8. **bisection ハーネス**。バグが 2 つの既知状態（commit、データセット、バージョン）の間で出るなら、「状態 X で起動 → チェック → 繰り返す」を自動化して `git bisect run` に乗せる
9. **差分ループ**。同じ入力を旧バージョン vs 新バージョン（または 2 つの config）に流し、出力を diff する
10. **HITL bash スクリプト**。最終手段。人間がクリックしないと駄目なら、`scripts/hitl-loop.template.sh` で**彼ら**を駆動し、ループは構造化を保つ。キャプチャした出力があなたへフィードバックされる

正しいフィードバックループを作れば、バグは 90% 治っている。

### ループそのものを反復改善する

ループをプロダクトとして扱え。**ある**ループが手に入ったら、こう問え:

- もっと速くできるか？（セットアップをキャッシュ、無関係な初期化をスキップ、テスト範囲を狭める）
- シグナルを鋭くできるか？（「クラッシュしなかった」ではなく特定の症状に assert）
- もっと決定論的にできるか？（時刻を固定、RNG を seed、ファイルシステムを隔離、ネットワークを凍結）

30 秒の flaky なループはループ無しと大差ない。2 秒の決定論的ループはデバッグの超能力だ。

### 非決定論的バグ

目標はきれいな再現ではなく**再現率を上げること**。トリガーを 100 回ループ、並列化、負荷をかける、タイミングウィンドウを狭める、sleep を挿入。50% flake のバグはデバッグできる。1% は無理 — デバッグ可能になるまで率を上げ続ける。

### 本当にループを作れないとき

止まれ、そして明示的にそう言え。試したことを列挙する。ユーザーに次のいずれかを求める: (a) 再現する環境へのアクセス、(b) キャプチャしたアーティファクト（HAR ファイル、ログダンプ、コアダンプ、タイムスタンプ付きスクリーン録画）、または (c) 本番に一時計装を入れる許可。ループ無しで仮説立てに進んで**はいけない**。

信頼できるループが無い限り Phase 2 に進まない。

## Phase 2 — 再現する

ループを走らせる。バグが現れるのを見る。

確認:

- [ ] ループは**ユーザーが**説明した failure mode を出している — 似たような別の失敗ではない。バグ違いは修正違いだ
- [ ] 失敗は複数回走らせて再現する（非決定論的バグの場合はデバッグに耐える率で再現する）
- [ ] 後のフェーズで修正が本当に効いたか検証できるよう、正確な症状（エラーメッセージ、間違った出力、遅延タイミング）をキャプチャしてある

バグを再現するまで先に進むな。

## Phase 3 — 仮説を立てる

どれも検証する前に、**ランク付けされた 3〜5 個の仮説**を生成する。単一仮説生成は最初に思いついた案にアンカーされる。

各仮説は**反証可能**でなければならない: 立てる予測を述べる。

> 形式: 「もし <X> が原因なら、<Y を変える> とバグが消える / <Z を変える> と悪化する」

予測を述べられない仮説は単なる気分 — 捨てるか研ぎ直す。

**ランク付けリストを検証前にユーザーに見せる。** ユーザーは即座にランクを並べ替えるドメイン知識を持っていることが多い（「#3 の変更を直前にデプロイした」）し、すでに除外した仮説を知っているかもしれない。安いチェックポイントで大きな時短になる。ユーザーが AFK ならブロックせず自分のランクで進む。

## Phase 4 — 計装する

各プローブは Phase 3 の特定の予測に対応していること。**変数は一度に 1 つだけ変える。**

ツールの優先度:

1. 環境が対応するなら**デバッガ / REPL 検査**。ブレークポイント 1 つはログ 10 個に勝る
2. 仮説を区別する境界での**ターゲットログ**
3. 「全部ログって grep」は決してやらない

**すべてのデバッグログにユニークな prefix をタグ付ける**。例: `[DEBUG-a4f2]`。最後のクリーンアップが grep 1 回で済む。タグ無しは生き残り、タグ付きは死ぬ。

**Perf 分岐。** performance regression ではログは大抵間違っている。代わりに: ベースライン計測（タイミングハーネス、`performance.now()`、プロファイラ、クエリプラン）を確立してから bisection。**測れ、その後に直せ。**

## Phase 5 — 修正 + regression test

regression test は**修正より先に**書く — ただし**正しい Seam** が存在する場合のみ。

正しい Seam とは、テストが call site で発生したそのままの**実バグパターン**を踏むこと。利用可能な唯一の Seam が浅すぎる（バグは複数呼び出しが必要なのに単一呼び出し用のテスト、バグを起こした連鎖を再現できない unit テスト）場合、そこに regression test を置くと偽の安心感を与えるだけだ。

**正しい Seam が無いこと自体が発見である。** メモする。コードベースのアーキテクチャがバグをロックダウンできないようにしている。次フェーズへフラグを立てる。

正しい Seam がある場合:

1. 最小化した再現をその Seam で失敗するテストにする
2. それが失敗するのを見る
3. 修正を当てる
4. それが通るのを見る
5. オリジナル（最小化前）シナリオに対して Phase 1 のフィードバックループを再実行する

## Phase 6 — クリーンアップ + ポストモーテム

完了を宣言する前に必須:

- [ ] オリジナル再現がもう再現しない（Phase 1 ループを再実行）
- [ ] regression test が通る（または Seam が無いことを文書化）
- [ ] すべての `[DEBUG-...]` 計装を除去済み（prefix を `grep`）
- [ ] 使い捨てプロトタイプを削除済み（または明示的にマークされたデバッグ場所へ移動）
- [ ] 正しかった仮説をコミット / PR メッセージに明記 — 次のデバッガが学べるように

**その上で問え: 何があればこのバグは防げたか？** 答えがアーキテクチャ変更（良いテスト Seam が無い、呼び出し関係が絡まっている、隠れた結合）を要するなら、具体を添えて `/improve-codebase-architecture` スキルへ引き渡す。修正が入った**あとに**推奨する — 始めたときより情報が増えている。
