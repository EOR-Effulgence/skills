# ADR フォーマット

ADR は `docs/adr/` に置き、連番で命名する: `0001-slug.md`, `0002-slug.md` ...

`docs/adr/` ディレクトリは**遅延作成** — 最初の ADR が必要になったときに作る。

## テンプレート

```md
# {決定の短いタイトル}

{1〜3 文: 文脈、何を決めたか、なぜか。}
```

これで終わり。ADR は 1 段落で構わない。価値は**決定がなされた事実**と**理由**を記録することにある。セクションを埋めることではない。

## オプションセクション

以下は本当に価値を加えるときだけ含める。ほとんどの ADR には不要。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`） — 決定が見直されるとき有用
- **Considered Options** — 却下した代替案を覚えておく価値があるときだけ
- **Consequences** — 自明でない下流影響を明示する必要があるときだけ

## 番号付け

`docs/adr/` 内の最大番号をスキャンして +1 する。

## ADR を提案するタイミング

以下の **3 つすべて** が真のときに限る:

1. **後戻りしづらい** — 後で気が変わったときのコストが意味のある大きさ
2. **文脈なしでは驚かされる** — 将来の読者がコードを見て「なぜこうしたんだ？」と思う
3. **本物のトレードオフの結果** — 真の代替案があり、特定の理由で選んだ

決定が簡単に後戻りできるなら省略する — どうせ後戻りする。驚きがなければ誰も理由を疑問に思わない。真の代替案がなかったなら「自明なことをやった」以上に記録すべきことはない。

### ADR に値するもの

- **アーキテクチャの形状。** 「モノレポを使う」「書き込みモデルはイベントソース、読み込みモデルは Postgres に投影」
- **コンテキスト間の統合パターン。** 「Ordering と Billing は同期 HTTP ではなくドメインイベントで通信する」
- **ロックインを伴う技術選択。** DB、メッセージバス、認証プロバイダ、デプロイ先。ライブラリ全部ではなく、置換に四半期かかるレベルのもの。
- **境界とスコープの決定。** 「Customer データは Customer コンテキストが所有する。他コンテキストは ID 参照のみ」 — 明示的な **no** は **yes** と同じくらい価値がある。
- **明示的な王道からの逸脱。** 「ORM ではなく手書き SQL を使う、理由は X」 — 妥当な読者が逆を仮定するもの。次のエンジニアが意図的だったものを「修正」するのを止める。
- **コードに現れない制約。** 「コンプライアンス要件で AWS が使えない」「パートナー API 契約でレスポンス < 200ms」
- **却下した代替案 — 却下理由が自明でないとき。** GraphQL を検討し微妙な理由で REST を選んだなら記録する。さもなくば 6 ヶ月後に誰かが再び GraphQL を提案する。
