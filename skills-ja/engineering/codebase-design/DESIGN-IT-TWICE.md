# Design It Twice

選ばれた deepening 候補について代替 Interface を探索したいとき、この並列サブエージェントパターンを使う。"Design It Twice"（Ousterhout）に基づく。最初のアイデアが最良である見込みは薄い。

[SKILL.md](SKILL.md) の語彙 — **module**、**interface**、**seam**、**adapter**、**leverage** — を使う。

## Process

### 1. 問題空間を枠づける

サブエージェントを立ち上げる前に、選ばれた候補について問題空間のユーザー向け説明を書く:

- 新しい Interface が満たすべき制約
- それが依拠する依存と、それがどのカテゴリに入るか（[DEEPENING.md](DEEPENING.md) を参照）
- 制約を地に足のついたものにするための、ざっくりした説明用コードスケッチ — 提案ではなく、制約を具体的にするための手段にすぎない

これをユーザーに見せ、直ちに Step 2 へ進む。サブエージェントが並列で作業する間、ユーザーはそれを読んで考える。

### 2. サブエージェントを立ち上げる

Agent tool を使って 3 つ以上のサブエージェントを並列で立ち上げる。それぞれが deepen した module に対して **極端に異なる** Interface を生み出さねばならない。

各サブエージェントには個別の technical brief（ファイルパス、結合の詳細、[DEEPENING.md](DEEPENING.md) の依存カテゴリ、Seam の背後に何があるか）を渡してプロンプトする。この brief は Step 1 のユーザー向け問題空間説明とは独立している。各エージェントに異なる設計制約を与える:

- Agent 1: 「Interface を最小化する — 入口を最大 1〜3 個に抑える。入口あたりの Leverage を最大化する。」
- Agent 2: 「柔軟性を最大化する — 多くのユースケースと拡張をサポートする。」
- Agent 3: 「最も一般的な呼び出し側に最適化する — デフォルトのケースを些細なものにする。」
- Agent 4（該当する場合）: 「Seam をまたぐ依存に対して ports & adapters を軸に設計する。」

brief には [SKILL.md](SKILL.md) の語彙と CONTEXT.md の語彙の両方を含め、各サブエージェントがアーキテクチャの言葉とプロジェクトのドメイン言語に一貫した名前で物事を呼ぶようにする。

各サブエージェントの出力:

1. Interface（型、メソッド、パラメータ — に加えて不変条件、順序、error mode）
2. 呼び出し側がどう使うかを示す使用例
3. Implementation が Seam の背後に何を隠すか
4. 依存戦略と Adapter（[DEEPENING.md](DEEPENING.md) を参照）
5. トレードオフ — どこで Leverage が高く、どこで薄いか

### 3. 提示して比較する

ユーザーが 1 つずつ吸収できるよう設計を順に提示し、その後で散文で比較する。**depth**（Interface における Leverage）、**locality**（変更が集中する場所）、そして **seam の配置** で対比する。

比較の後、自分自身の推奨を述べる。どの設計が最も強いと考えるか、そしてなぜか。異なる設計の要素がうまく組み合わさりそうなら、ハイブリッドを提案する。はっきり意見を持て — ユーザーはメニューではなく、強い読みを求めている。
