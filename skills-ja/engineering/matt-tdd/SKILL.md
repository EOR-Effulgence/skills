# テスト駆動開発 (TDD)

## 哲学

**中核原則**: テストは**実装の詳細**ではなく、**公開インターフェース経由のふるまい**を検証する。コードは全面的に書き換えてよい。テストは書き換えないで済むべきだ。

**良いテスト**は統合的（integration-style）。公開 API 経由で実際のコードパスを走らせ、システムが「**何を**するか」を記述する。「**どう**やるか」は書かない。良いテストは仕様書のように読める — `user can checkout with valid cart` と書けば、その能力が存在することが一目で分かる。内部構造を気にしないため、リファクタを生き延びる。

**悪いテスト**は実装に密結合している。内部の協力者をモックし、private メソッドをテストし、インターフェースを経由せずに（DB を直接クエリして）検証する。警告サイン: **ふるまいは変わっていないのにリファクタでテストが壊れる**。内部関数をリネームしただけでテストが落ちるなら、それは実装をテストしていたのであってふるまいではない。

例は [tests.md](tests.md)、モック方針は [mocking.md](mocking.md) を参照。

## アンチパターン: 水平スライス (Horizontal Slices)

**全テストを先に書いてから全実装を書く、これは絶対にやらない。** これは「水平スライス」と呼ばれ、RED を「全テスト書く」、GREEN を「全コード書く」と誤解した状態だ。

これはクソテスト（crap tests）を生む:

- 一括で書いたテストは**想像上の**ふるまいをテストし、**実際の**ふるまいをテストしない
- データ構造や関数シグネチャといった**形状（shape）**をテストすることになり、ユーザー視点のふるまいをテストしない
- 実際の変化に対して鈍感になる — ふるまいが壊れても通り、ふるまいが正しくても落ちる
- 実装を理解する前にテスト構造にコミットしてしまう（ヘッドライトより先に走る）

**正しいアプローチ**: トレーサー弾（tracer bullets）による垂直スライス。テスト 1 個 → 実装 1 個 → 繰り返す。各テストは前サイクルで学んだことに応答する。コードを書いた直後なので、どのふるまいが重要で、どう検証すべきかが正確に分かる。

```
誤り (水平):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正しい (垂直):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## ワークフロー

### 1. 計画

コードベースを探索する際は、プロジェクトの**ドメイン用語集**を使い、テスト名やインターフェースの語彙をプロジェクト言語と一致させる。触る領域の ADR も尊重する。

コードを書き始める前に:

- [ ] どのインターフェースを変更するか、ユーザーと合意
- [ ] どのふるまいをテストするか、優先順位をユーザーと合意
- [ ] **深いモジュール**（小さなインターフェース + 厚い実装）の機会を特定 → [deep-modules.md](deep-modules.md)
- [ ] **テスト可能性**を考えたインターフェース設計 → [interface-design.md](interface-design.md)
- [ ] テスト対象のふるまいを列挙（実装ステップではない）
- [ ] 計画の承認をユーザーから得る

問いかけ: 「公開インターフェースはどうあるべきですか？ どのふるまいが最も重要ですか？」

**すべてはテストできない。** 何が最重要かをユーザーに必ず確認する。クリティカルパスと複雑なロジックに集中し、あらゆるエッジケースを網羅しようとしない。

### 2. トレーサー弾

システムについて**ひとつのこと**を確認する**ひとつのテスト**を書く:

```
RED:   最初のふるまいのテストを書く → 落ちる
GREEN: 通すための最小コードを書く → 通る
```

これがトレーサー弾。経路が端から端まで通ることを証明する。

### 3. 増分ループ

残りのふるまいそれぞれについて:

```
RED:   次のテストを書く → 落ちる
GREEN: 通すための最小コード → 通る
```

ルール:

- 一度に 1 テスト
- 現在のテストを通すのに必要なコードだけ
- 将来のテストを先取りしない
- 観測可能なふるまいにテストの焦点を当て続ける

### 4. リファクタ

すべてのテストが通ったら、[リファクタ候補](refactoring.md)を探す:

- [ ] 重複の抽出
- [ ] モジュールの深化（複雑性をシンプルなインターフェースの裏に隠す）
- [ ] 自然な箇所で SOLID 原則を適用
- [ ] 新コードが既存コードについて何を明らかにしたか考える
- [ ] 各リファクタ後にテストを走らせる

**RED の状態でリファクタしてはいけない。** まず GREEN にする。

## サイクルごとのチェックリスト

```
[ ] テストはふるまいを記述している（実装ではない）
[ ] テストは公開インターフェースのみを使う
[ ] テストは内部リファクタを生き延びる
[ ] このテストにとって最小のコードになっている
[ ] 投機的な機能を追加していない
```
