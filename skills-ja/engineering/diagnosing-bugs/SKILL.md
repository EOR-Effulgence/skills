# Diagnosing Bugs

難しいバグのための規律。明示的に正当化できるときだけフェーズをスキップしろ。

コードベースを探索する際は `CONTEXT.md`（存在すれば）を読んで関連 Module の明確なメンタルモデルを得て、触る領域の ADR を確認する。

## Phase 1 — フィードバックループを作る

**ここがこのスキルの本体だ。** ほかは全部メカニカルでしかない。バグに対する**タイト**な pass/fail signal — _この_バグで赤くなるもの — を持っているなら、原因は必ず見つかる。bisection、hypothesis testing、計装はすべてそれを消費しているだけだ。持っていないなら、コードをいくら睨んでも救われない。

ここに不釣り合いなほど労力を注ぐ。**攻めろ。創造的になれ。あきらめるな。**

### 作り方 — おおむねこの順で試す

1. バグに届く seam での**失敗するテスト** — unit、integration、e2e
2. 動いている dev サーバーに対する **curl / HTTP スクリプト**
3. fixture 入力を渡す **CLI 起動**。既知の良い snapshot に対して stdout を diff する
4. **ヘッドレスブラウザスクリプト**（Playwright / Puppeteer）— UI を駆動し、DOM / コンソール / ネットワークに assert する
5. **キャプチャしたトレースをリプレイ。** 実ネットワークリクエスト / ペイロード / イベントログをディスクに保存し、独立にコード経路へ流し直す
6. **使い捨てハーネス。** システムの最小サブセット（1 サービス、依存はモック）を立ち上げ、関数 1 回の呼び出しでバグのコード経路を踏むようにする
7. **property / fuzz ループ。** バグが「たまに間違った出力」型なら、ランダム入力 1000 件を流して failure mode を探す
8. **bisection ハーネス。** バグが 2 つの既知状態（commit、データセット、バージョン）の間で出るなら、「状態 X で起動 → チェック → 繰り返す」を自動化して `git bisect run` に乗せる
9. **differential ループ。** 同じ入力を旧バージョン vs 新バージョン（または 2 つの config）に流し、出力を diff する
10. **HITL bash スクリプト。** 最終手段。人間がクリックしないと駄目なら、`scripts/hitl-loop.template.sh` で**彼ら**を駆動し、ループは構造化を保つ。キャプチャした出力があなたへフィードバックされる

正しいフィードバックループを作れば、バグは 90% 治っている。

### ループをタイトにする

ループをプロダクトとして扱え。**ある**ループが手に入ったら、それを**タイトにする**:

- もっと速くできるか？（セットアップをキャッシュ、無関係な初期化をスキップ、テスト範囲を狭める）
- signal をもっと鋭くできるか？（「クラッシュしなかった」ではなく特定の症状に assert する）
- もっと決定論的にできるか？（時刻を固定、RNG を seed、ファイルシステムを隔離、ネットワークを凍結）

30 秒の flaky なループはループ無しと大差ない。2 秒の決定論的ループはタイトだ — デバッグの超能力だ。

### 非決定論的バグ

目標はきれいな再現ではなく、**再現率を上げること**だ。トリガーを 100 回ループし、並列化し、負荷をかけ、タイミングウィンドウを狭め、sleep を挿入する。50% flake のバグはデバッグできる。1% は無理 — デバッグ可能になるまで率を上げ続ける。

### 本当にループを作れないとき

止まれ、そして明示的にそう言え。試したことを列挙する。ユーザーに次のいずれかを求める: (a) 再現する環境へのアクセス、(b) キャプチャしたアーティファクト（HAR file、ログダンプ、コアダンプ、タイムスタンプ付きスクリーン録画）、または (c) 本番に一時計装を入れる許可。ループ無しで仮説立てに進んで**はいけない**。

### 完了基準 — 赤くなれるタイトなループ

Phase 1 は、ループが**タイト**かつ**赤くなれる**状態になったら完了だ。**すでに最低 1 回は実行した** **1 つのコマンド** — スクリプトパス、テスト起動、curl のいずれか — を名指しでき（起動コマンドとその出力を貼る）、それが次を満たすこと:

- [ ] **赤くなれる** — 実際のバグのコード経路を駆動し、**ユーザーの正確な症状**に assert する。だからこのバグで赤くなれ、直れば緑になれる。「エラーせず走る」ではない — _この特定のバグを捕まえられる_こと
- [ ] **決定論的** — 毎回同じ判定（flaky バグの場合: 上記のとおり固定された高い再現率）
- [ ] **高速** — 分ではなく秒
- [ ] **エージェントが実行可能** — 無人で走らせられる。ループ内の人間は `scripts/hitl-loop.template.sh` 経由のみ

このコマンドが存在する前に理論を組み立てようとコードを読み始めている自分に気づいたら、**止まれ — いきなり仮説へ飛ぶことこそ、このスキルが防ぐまさにその失敗だ。** 赤くなれるコマンドが無ければ、Phase 2 は無い。

## Phase 2 — 再現 + 最小化

ループを走らせる。赤くなるのを見る — バグが現れる。

確認:

- [ ] ループは**ユーザー**が説明した failure mode を出している — たまたま近くにある別の失敗ではない。バグ違いは修正違いだ
- [ ] 失敗は複数回の実行で再現する（非決定論的バグの場合は、デバッグに耐えるだけの十分な率で再現する）
- [ ] 後のフェーズで修正が本当にそれに効いたか検証できるよう、正確な症状（エラーメッセージ、間違った出力、遅延タイミング）をキャプチャしてある

### 最小化

赤くなったら、再現を**まだ赤くなる最小のシナリオ**まで縮める。入力、呼び出し元、config、データ、ステップを**一度に 1 つずつ**削り、削るたびにループを再実行する — 失敗を支えているものだけを残す。

なぜやるか: 最小の再現は Phase 3 の仮説空間を縮め（残る疑うべき可動部が減る）、Phase 5 でのきれいな regression test になる。

**残った要素がすべて失敗を支えている**とき完了 — どれ 1 つ取り除いてもループが緑になる状態だ。

再現**かつ**最小化を済ませるまで先に進むな。

## Phase 3 — 仮説を立てる

どれも検証する前に、**ランク付けされた 3〜5 個の仮説**を生成する。単一仮説生成は最初に思いついた尤もらしい案にアンカーされる。

各仮説は**反証可能**でなければならない: 立てる予測を述べる。

> 形式: 「もし <X> が原因なら、<Y を変える> とバグが消える / <Z を変える> と悪化する」

予測を述べられないなら、その仮説は単なる気分だ — 捨てるか研ぎ直す。

**ランク付けリストを検証前にユーザーに見せる。** ユーザーは即座にランクを並べ替えるドメイン知識を持っていることが多く（「#3 への変更をちょうどデプロイした」）、あるいはすでに除外した仮説を知っているかもしれない。安いチェックポイントで大きな時短になる。ただしそこでブロックはするな — ユーザーが AFK なら自分のランクで進む。

## Phase 4 — 計装する

各プローブは Phase 3 の特定の予測に対応していること。**変数は一度に 1 つだけ変える。**

ツールの優先度:

1. 環境が対応するなら**デバッガ / REPL 検査**。ブレークポイント 1 つはログ 10 個に勝る
2. 仮説を区別する境界での**ターゲットログ**
3. 「全部ログって grep」は決してやらない

**すべてのデバッグログにユニークな prefix をタグ付けろ**。例: `[DEBUG-a4f2]`。最後のクリーンアップが grep 1 回で済む。タグ無しは生き残り、タグ付きは死ぬ。

**Perf 分岐。** performance regression では、ログは大抵間違っている。代わりに: ベースライン計測（タイミングハーネス、`performance.now()`、プロファイラ、クエリプラン）を確立してから bisection する。先に測れ、直すのはその後だ。

## Phase 5 — 修正 + regression test

regression test は**修正より先に**書く — ただし**正しい seam** が存在する場合のみ。

正しい seam とは、テストが call site で発生したそのままの**実バグパターン**を踏むものだ。利用可能な唯一の seam が浅すぎる場合（バグは複数の呼び出し元が必要なのに単一呼び出し元用のテスト、バグを起こした連鎖を再現できない unit テスト）、そこに置いた regression test は偽の安心感を与える。

**正しい seam が無いこと自体が発見だ。** メモする。コードベースのアーキテクチャがバグをロックダウンできないようにしている。これを次フェーズへフラグとして立てる。

正しい seam がある場合:

1. 最小化した再現を、その seam で失敗するテストにする
2. それが失敗するのを見る
3. 修正を当てる
4. それが通るのを見る
5. オリジナル（最小化前）のシナリオに対して Phase 1 のフィードバックループを再実行する

## Phase 6 — クリーンアップ + ポストモーテム

完了を宣言する前に必須:

- [ ] オリジナルの再現がもう再現しない（Phase 1 ループを再実行）
- [ ] regression test が通る（または seam が無いことを文書化）
- [ ] すべての `[DEBUG-...]` 計装を除去済み（prefix を `grep`）
- [ ] 使い捨てプロトタイプを削除済み（または明示的にマークされたデバッグ場所へ移動）
- [ ] 正しかった仮説をコミット / PR メッセージに明記 — 次のデバッガが学べるように

**その上で問え: 何があればこのバグは防げたか？** 答えがアーキテクチャ変更（良いテスト seam が無い、呼び出し元が絡まっている、隠れた結合）を要するなら、具体を添えて `/improve-codebase-architecture` スキルへ引き渡す。推奨は修正が入った**あと**に出せ、前ではない — 始めたときより今のほうが情報が多い。
