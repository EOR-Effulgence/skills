# Prototype

プロトタイプとは、**問いに答えるための使い捨てコード**だ。問いが形を決める。

## 分岐を選ぶ

ユーザーのプロンプト、周辺コード、または（連絡が取れるなら）ユーザーへの確認から、どの問いに答えているのかを特定する:

- **「このロジック / 状態モデルは正しい感じがするか？」** → [LOGIC.md](LOGIC.md)。紙の上では考えにくいケースに状態マシンを通してみる、小さなインタラクティブなターミナルアプリを作る。
- **「これはどう見えるべきか？」** → [UI.md](UI.md)。1 つのルート上に極端に異なる UI バリアントを複数生成し、URL クエリパラメータとフローティングのボトムバーで切り替える。

2 つの分岐は全く異なる成果物を生む — 間違えるとプロトタイプ全体が無駄になる。問いが本当に曖昧でユーザーに連絡が取れない場合は、周辺コードに合わせる方を選び（バックエンド Module → logic、ページやコンポーネント → UI）、その仮定をプロトタイプの先頭に明記する。

## 両分岐に共通するルール

1. **初日から使い捨てであり、そうと明示する。** プロトタイプは実際に使われる場所（プロトタイプ対象の Module・ページの隣）に置きコンテキストを明らかにする — ただし読み手がプロトタイプだと分かる名前を付ける。使い捨ての UI ルートはプロジェクトの既存ルーティング規約に従い、新しいトップレベル構造を発明しない。
2. **1 コマンドで動く。** プロジェクトの既存タスクランナーが対応するもの — `pnpm <name>`、`python <path>`、`bun <path>` など。ユーザーが何も考えずに起動できなければならない。
3. **デフォルトで永続化なし。** 状態はメモリに置く。永続化はプロトタイプが**検証している対象**であり、依存するものではない。問いが明示的に DB を含むなら、「PROTOTYPE — wipe me」のような明確な名前のスクラッチ DB かローカルファイルを使う。
4. **磨きをスキップする。** テストなし、エラーハンドリングはプロトタイプを**動かす**のに必要な最小限のみ、抽象化なし。要点は速く何かを学んで捨てることだ。
5. **状態を見せる。** あらゆるアクションのあと（logic）、またはバリアント切り替えのたび（UI）、関連する全状態を表示・レンダリングして変化が見えるようにする。
6. **済んだら削除または吸収する。** プロトタイプが問いに答え終わったら、削除するか、検証済みの判断を本物のコードに畳み込む — repo に腐らせない。

## 終わったら

プロトタイプから残す価値があるのは**答え**だけだ。それを耐久的な場所（コミットメッセージ、ADR、Issue、プロトタイプの隣の `NOTES.md`）に、答えた問いと一緒に記録する。ユーザーが居るならその記録は短い会話で済む。居ないなら、プレースホルダーを残し、削除する前に（彼らか、次に通るあなた自身が）結論を埋められるようにする。
