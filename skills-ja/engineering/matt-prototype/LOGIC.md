# Logic Prototype

ユーザーが手で状態モデルを駆動できる、小さなインタラクティブなターミナルアプリ。問いが**ビジネスロジック、状態遷移、データ形状**に関するもの — 紙の上では理にかなって見えるが、実ケースを通すと違和感が出るもの — に対して使う。

## これが正しい形状になるとき

- 「この状態マシンが X のあとに Y が来るエッジケースを処理できるか自信がない」
- 「このデータモデルは実際に … のケースを表現できるか」
- 「書く前に API のあるべき姿を触って感じたい」
- ユーザーが**ボタンを押して状態の変化を見たい**もの全般。

問いが「これはどう見えるべきか」なら分岐違い。[UI.md](UI.md) を使う。

## プロセス

### 1. 問いを明文化する

コードを書く前に、何の状態モデル・何の問いをプロトタイプしているかを書き留める。1 段落でいい。プロトタイプの README か、ファイル先頭のコメントに書く。間違った問いに答える logic プロトタイプは純粋な無駄 — 問いを明示しておけば後で確認できる。ユーザーが今見ていようと、AFK で戻ってこようと変わらない。

### 2. 言語を選ぶ

ホストプロジェクトが使っているものを使う。プロジェクトに明確なランタイムが無い場合（docs リポジトリなど）は確認する。

ツーリングはプロジェクトの既存規約に合わせる — プロトタイプのために新しいパッケージマネージャやランタイムを足さない。

### 3. ロジックを移植可能な Module に隔離する

問いに答えている実ロジックを、小さく純粋なインターフェースの背後に置き、後で本物のコードベースへそのまま持っていけるようにする。周りの TUI は使い捨て。ロジック Module はそうではない。

正しい形は問いによって変わる:

- **純粋なリデューサー** — `(state, action) => state`。アクションが離散的イベントで、状態が単一の値のときに良い
- **状態マシン** — 明示的な状態と遷移。「今このアクションを実行してよいか」が問いの一部のときに良い
- **平らなデータ型に対する純粋関数の小さなセット**。暗黙の現在状態がなく、変換だけのときに良い
- **明確なメソッド面を持つクラスや Module**。ロジックが本当に継続的な内部状態を所有するとき

問いに最も合う形を選ぶ。TUI に繋ぐのが最も楽な形ではない。純粋に保つこと: I/O なし、ターミナルコードなし、制御フローのための `console.log` なし。TUI が import して呼ぶ。逆方向には何も流さない。

これがプロトタイプを自身の寿命を超えて有用にする鍵だ。問いに答え終わったら、検証済みのリデューサー / マシン / 関数群を本物の Module へ持ち上げ、TUI のシェルは削除する。

### 4. 状態を見せる最小の TUI を作る

**軽量 TUI** として作る — 毎ティック画面をクリア（`console.clear()` / `print("\033[2J\033[H")` / 同等品）し、フレーム全体を再描画する。ユーザーには常に 1 つの安定したビューを見せ、スクロールバックが膨らみ続けないようにする。

各フレームは次の順序の 2 部構成:

1. **現在の状態**、見やすく diff フレンドリーに（1 行 1 フィールド、または整形済み JSON）。フィールド名やセクション見出しは **bold**、重要度の低いコンテキスト（タイムスタンプ、ID、派生値）は **dim**。生の ANSI エスケープで構わない — `\x1b[1m` bold、`\x1b[2m` dim、`\x1b[0m` reset。プロジェクトに既に入っているのでなければスタイリングライブラリを引っ張ってこなくていい
2. **キーボードショートカット**を下部に列挙: `[a] add user  [d] delete user  [t] tick clock  [q] quit`。キーを bold、説明を dim、または逆 — 読みやすい方で

挙動:

1. **状態を初期化** — メモリ上の単一のオブジェクト / 構造体。起動時に最初のフレームを描画する
2. **キーストローク（または 1 行）を 1 つずつ読み**、状態を変更するハンドラへディスパッチ
3. アクションのたびに**フレーム全体を再描画** — 追記ではなく差し替え
4. **quit まで loop**

フレーム全体は 1 画面に収まること。

### 5. 1 コマンドで動くようにする

プロジェクト既存のタスクランナー（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）にスクリプトを足す。ユーザーは `pnpm run <prototype-name>` 等で実行する — パスを覚える必要は無い。

ホストプロジェクトにタスクランナーが無いなら、コマンドをプロトタイプの README 先頭に書いておく。

### 6. 引き渡す

ユーザーに実行コマンドを渡す。あとは彼ら自身で駆動する。面白い瞬間は「待って、こうなるはずがない」「あれ、X はこうじゃないと思ってた」と言う瞬間 — それが**アイデア**そのもののバグであり、これこそが要点だ。新しいアクションを足したいと言われたら足す。プロトタイプは進化する。

### 7. 答えを記録する

プロトタイプが仕事を終えたら、問いへの答えだけが残す価値のあるもの。ユーザーが居るなら何を学んだか聞く。居ないなら、プロトタイプの隣に `NOTES.md` を残し、削除前に答えを埋められるようにする（セッションを観察したならあなたが埋めてもよい）。

## アンチパターン

- **テストを足すな。** テストが必要なプロトタイプはもうプロトタイプではない。
- **本物の DB に繋ぐな。** 問いが明示的に永続化の話でない限り、インメモリストアを使う。
- **一般化するな。** 「将来 X もサポートしたくなったら」は禁句。プロトタイプは 1 つの問いに答える。
- **ロジックと TUI を混ぜるな。** リデューサー / 状態マシンが `console.log`、prompt、ターミナルエスケープを参照していたら、もう移植可能ではない。TUI は純粋 Module の上の薄いシェルに留める。
- **TUI シェルを本番に出すな。** シェルはターミナルから手で叩かれることに最適化されている。残す価値があるのはその裏のロジック Module だ。
