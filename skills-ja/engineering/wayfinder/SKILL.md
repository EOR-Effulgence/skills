漠然としたアイデアが届いた — 一つのエージェントセッションには大きすぎ、霧に包まれている。ここから**目的地**への道はまだ見えない。Wayfinding とは、目的地に突進することではなく、その道を見つけることだ。このスキルは、その道をリポジトリの issue tracker 上の**共有マップ**として描き出し、その **decision ticket** — 解決が「決定」であってビルドの実行スライスではない問い — を、道筋が明確になるまで一つずつ処理する。

目的地は取り組みごとに異なり、それに名前を付けることが charting の最初の行為だ — それがすべてのチケットを形づくる。目的地は、引き渡して反復する仕様かもしれないし、計画を始める前に固める決定かもしれないし、データ構造の migration のようにその場で加える変更かもしれない。マップはドメイン非依存だ — エンジニアリング作業でも、コース教材でも、この形に合うものなら何でもよい。

## 計画せよ、実行するな

Wayfinder はデフォルトで**計画**だ。各チケットは一つの決定を解決し、道が明確になった時点でマップは完成する — 誰かが実際に手を動かす前に決めるべきことが何も残っていない状態だ。とにかく作業を進めたいという引力は、たいていマップの端に達したというサインであり、引き渡しの時が来たということだ。取り組みによっては **Notes** でこれを上書きし、実行そのものをマップに取り込むこともできる。だがそうでない限り、成果物ではなく決定を生み出せ。

## 名前で参照する

すべてのマップとチケットは issue なので、**名前** — つまりタイトル — を持つ。人間が読むものすべて — ナレーション、マップの Decisions-so-far — では、その名前で参照し、素の id・番号・slug では決して参照するな。`#42, #43, #44` の羅列は判読できない。名前なら一目で読める。id と URL が消えるわけではない — 名前がそのリンクを包む — が、それらは名前の*内側*に乗るのであって、名前の代わりにはならない。

## マップ

マップは、このリポジトリの issue tracker 上の単一の issue で、`wayfinder:map` ラベルが付く — 正典となる成果物だ。そのチケット群はマップの子 issue である。

マップは**インデックス**であって、格納庫ではない。下された決定を列挙し、その詳細を保持するチケットを指す。決定はただ一つの場所 — そのチケット — にのみ存在するので、マップは決定を再掲せず、要点だけを示してリンクする。

**マップ・その子チケット・ブロック・フロンティアクエリが物理的にどこに存在するかは tracker 依存だ。** issue tracker はすでに提供されているはず — もしなければ `/setup-matt-pocock-skills` を実行せよ。_この_リポジトリがそれらをどう表現するかは、tracker ドキュメントの "Wayfinding operations" セクションを参照せよ。tracker が提供されていなければ、local-markdown tracker をデフォルトとする。

### マップ本体

マップ全体を低解像度で表したもので、セッションごとに一度だけ読み込む。オープンなチケットは**列挙されない** — それらはオープンな子 issue であり、クエリで見つける。

```markdown
## Destination

<what reaching the end of this map looks like — the spec, decision, or change this effort is finding its way to. One or two lines; every session orients to it before choosing a ticket.>

## Notes

<domain; skills every session should consult; standing preferences for this effort>

## Decisions so far

<!-- the index — one line per closed ticket: enough to judge relevance, then zoom the link for the detail the ticket holds -->

- [<closed ticket title>](link) — <one-line gist of the answer>

## Not yet specified

<!-- see "Fog of war": in-scope fog you can't ticket yet; graduates as the frontier advances -->

## Out of scope

<!-- see "Out of scope": work ruled beyond the destination; closed, never graduates -->
```

### チケット

各チケットはマップの**子 issue** であり、tracker の issue id がその同一性を表す。本体は問いであり、100K token のエージェントセッション一つに収まるサイズにする。

```markdown
## Question

<the decision or investigation this ticket resolves>
```

各チケットは `wayfinder:<type>` ラベルを持つ — `research`・`prototype`・`grilling`・`task` のいずれか([Ticket Types](#ticket-types) を参照)。

セッションはチケットを**確保**する — マップを進めている開発者にそれをアサインすることで、しかも他のいかなる作業よりも**先に**行う。そうすれば並行するセッションはそれをスキップする。そのアサイン先_こそが_確保の証だ。オープンかつ未アサインのチケットは未確保である。

ブロックには tracker の**ネイティブな**依存関係を使う — これは不可欠だ。なぜなら tracker 自身の UI 上でフロンティアを_視覚的に_描き出し、人間がマップを開かずに何が取れるかを見られるからだ。ネイティブなブロックを欠く tracker だけが、本体上の記法にフォールバックする。あるチケットをブロックしているチケットがすべてクローズされたとき、そのチケットは**アンブロック**される。**フロンティア**とは、オープンでアンブロックかつ未確保の子チケット — 既知の縁 — だ。

答えは本体の一部ではない — 解決時に記録される([Work through the map](#work-through-the-map) を参照)。チケットを解決する過程で作られたアセットは、貼り付けるのではなく issue からリンクする。

## Ticket Types

すべてのチケットは **HITL**(human in the loop、自ら発言する人間と*ともに*進める)か、**AFK**(エージェント単独で進める)のいずれかだ。HITL チケットはそのライブなやり取りを通じてのみ解決する。エージェントが人間の側を代弁することは決してない(自分の問いに自分で答える grilling エージェントは、これを破っている)。

- **Research** (AFK): ドキュメント・サードパーティ API・ナレッジベースのようなローカルリソースを読み、決定が待っている事実を浮かび上がらせる。`/research` **subagent** が解決する。現在の作業ディレクトリの外の知識が必要なときに使う。
- **Prototype** (HITL): 反応の対象となる安価で粗い具体的な成果物 — アウトライン、ざっくりした試案、スタブ、または /prototype スキルによる UI/ロジックのコード — を作り、議論の解像度を上げる。プロトタイプはアセットとしてリンクする。「どう見えるべきか」「どう振る舞うべきか」が核心的な問いのときに使う。
- **Grilling** (HITL): /grilling と /domain-modeling スキルによる会話で、一度に一つの問いを扱う。デフォルトのケース。
- **Task** (HITL または AFK): *決定*を下す前に済ませておかなければならない手作業 — 決めること・プロトタイプ・リサーチは何もないが、それが終わるまで議論がブロックされる。API を評価できるようサービスにサインアップする、アクセスを払い出す、形を見られるようデータを移す、といったもの。これは決定するのではなく*実行する*唯一の種別だ — そして目的地を届けることによってではなく、決定のブロックを解除することによって存在意義を得る。可能な場合はエージェントが単独で進め(AFK)、そうでなければ人間に正確なチェックリストを手渡す(HITL)。作業が完了した時点で解決とする。答えには、実施した内容と、後続チケットが依存する結果的な事実(認証情報の場所、新しい URL、行数)を記録する。

## Fog of war

マップは_意図的に_不完全だ。まだ見えないものを描くな。生きているチケットの向こうには **fog of war**(戦場の霧)が広がる — 来ることは分かっていても、まだ開いている問いに依存しているために、まだ特定しきれない決定や調査のぼんやりとした眺めだ。チケットを解決するとその前方の霧が晴れ、いま特定可能になったものが新しいチケットへと昇格する — 一度に一つずつ、目的地への道が明確になりチケットが一つも残らなくなるまで。

マップの **Not yet specified** セクションは、そのぼんやりした眺めを書き留める場所だ。疑わしい問い、後で再訪すべき領域。それは目的地_へ向かう_未発見のフロンティアだ — ここにあるものはすべてスコープ内であり、ただチケットにできるほど鋭くないだけだ。眺めが許す範囲で、緩くでも詳しくでも書け。取り組みがどこへ向かっているかを読む協力者への道標も兼ねる。

**霧かチケットか?** 判定基準は、いま問いを正確に述べられるかどうかであって、いまそれに答えられるかどうか_ではない_。

- **チケットにするのは**、問いがすでに鋭いとき — たとえブロックされていて、まだ手を付けられなくても。
- **Not yet specified にするのは**、まだそこまで鋭く言い表せないとき。霧をチケットサイズの断片にあらかじめ切り分けるな。それはチケットより粗く、一つのまとまりはフロンティアが到達した時点で複数のチケットに昇格することも、一つも昇格しないこともある。

**Not yet specified** は、すでに決定済みのもの(Decisions so far)、すでに生きているチケット、スコープ外のもの(次のセクション)を除外する。

## Out of scope

霧は常に目的地_へ向かって_のみ集まる。目的地がスコープを固定するので、その先の作業は**スコープ外**だ — それは霧ではなく、**Not yet specified** には属さない。それはマップ上に独自の **Out of scope** セクションを持つ。_この_取り組みから意識的に除外した作業だ。ここに置かれる理由は鋭さではなくスコープだ。

スコープ外の作業は決して昇格しない — フロンティアは目的地で止まる — ので、それが戻ってくるのは目的地が描き直されたときだけであり、そのときも再開ではなく新しい取り組みとして戻る。

何かをスコープ外と決めることは、スコープを定める行為であって、ルート上の一歩ではない。すでに存在するチケットが目的地の先にあると判明したとき — charting 中に誤ってスコープに入れた、または解決によって露呈した — **それをクローズせよ**(クローズされたチケットは明白にフロンティアの外だ)。そして **Out of scope** セクションに一行だけ残す。要点と、なぜスコープ外なのか、そしてクローズしたチケットへのリンク。それは **Decisions so far** には入れない。Decisions so far は実際に歩いたルートを記録するものであり、スコープ境界はその一歩ではないからだ。

## 呼び出し

モードは二つ。いずれの場合も、**1 セッションで複数のチケットを解決するな** — research チケットは例外とする。

### マップを描く

ユーザーは漠然としたアイデアとともに呼び出す。

1. **目的地に名前を付ける。** `/grilling` と `/domain-modeling` のセッションを実行し、このマップが道を探している先 — 仕様・決定・変更 — を突き止める。目的地がスコープを固定するので、最初に確定させる。
2. **フロンティアをマップ化する。** 再び grill するが、今度は**幅優先**で。どれか一本の筋を深掘りするのではなく空間全体に扇状に広げ、開いている決定と、いま踏み出せる最初のステップを浮かび上がらせる。**これで霧が一つも出てこないなら** — 目的地への道はすでに明確で、旅全体が 1 セッションに収まるほど小さい — マップは要らない。そこで止まり、どう進めたいかをユーザーに尋ねる。
3. **マップを作成する**(ラベル `wayfinder:map`): Destination と Notes を埋め、Decisions-so-far は空、霧は **Not yet specified** に素描する。
4. **いま特定できるチケットを作成する** — マップの子 issue として — そして **second pass** でブロックのエッジを配線する(issue が互いを参照するには先に id が必要だ)。配線によってチケットはフロンティアとブロック済みに仕分けられる。まだ特定できないものはすべて霧 — **Not yet specified** セクション — に留まる。
5. **research subagent を起動する。** いま作成した各 `research` チケットについて、`/research` subagent を立ち上げて並列に解決し、その調査結果を使い捨ての `research/<name>` branch に取り込み、チケットからの context pointer を残す。
6. 止まる — charting は 1 セッション分の作業だ。手作業では何も解決しない。

### Work through the map

ユーザーはマップ(URL または番号)とともに呼び出す。チケットの指定は**任意**だ — 指定がなければ、ユーザーではなくあなたが次の決定を選ぶ。

1. **マップ**を読み込む — 低解像度の眺めであって、全チケットの本体ではない。
2. チケットを選ぶ。ユーザーが指定したならそれを使う。そうでなければ、順番で最初のフロンティアチケットを取る。**確保する**: 作業に入る前に自分自身にアサインする。
3. 解決する — **必要に応じてズームする**: 関連チケットやクローズ済みチケットの全文を必要に応じて取得し、`## Notes` ブロックが挙げるスキルを呼び出す。迷ったら `/grilling` と `/domain-modeling` を使う。
4. 解決を記録する: 答えを**解決コメント**として投稿し、issue を**クローズ**し、マップの Decisions-so-far に **context pointer を追記する**。
5. 新たに浮かび上がったチケットを追加する(作成してから配線)。答えによって特定可能になった霧を昇格させ、昇格した各まとまりを **Not yet specified** から消して、新しいチケットとしてのみ存在させる。もし答えによって、あるチケット — このチケットか別のか — が目的地の先にあると分かったら、ルート上で解決するのではなく**スコープ外と決める**。もしその決定がマップの他の部分を無効にするなら、それらのチケットを更新または削除する。

ユーザーはブロックされていないチケットを並列に実行するかもしれないので、他のセッションが tracker を同時に編集していることを想定せよ。
